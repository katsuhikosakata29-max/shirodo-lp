# /unknowns — 作業前の未知棚卸し

「地図は領土ではない」。プロンプト（地図）と実際のコードベース・現実の制約（領土）のズレ＝未知を、実装コストが高くなる前に発見するためのスキル。

## 手順

### 1. コンテキストの把握

まずユーザーのタスクと現在地を確認する。不明なら1問だけ聞く：

- 何を作ろうとしているか / 変えようとしているか
- その領域（技術・ドメイン）の経験はどの程度か

必要ならコードベースを軽く調査して、タスクが触る範囲を把握する。

### 2. 4象限への仕分け

タスクに関わる事項を以下の4象限に仕分けし、表で提示する：

| 象限 | 定義 | 例 |
|---|---|---|
| 既知の既知 | ユーザーが既に指示・決定済みのこと | 要件として明言されたこと |
| 既知の未知 | まだ決めていないと自覚していること | 「配色はまだ決めてない」 |
| 未知の既知 | 言語化していないが、見れば良し悪しが分かること | デザインの好み、UXの感覚 |
| 未知の未知 | 考慮すらしていない可能性があること | 運用、SEO、エッジケース |

### 3. 象限ごとのアクション実行

仕分け結果に応じて、該当する手法をこの場で実行する（別スキルは呼ばない。以下は本スキル内の動作モード）：

**既知の未知 → インタビュー**
曖昧な点を1問ずつ質問する（AskUserQuestion を使う）。**回答によって設計・アーキテクチャが変わる質問を優先**し、些末な質問は後回しにする。3〜5問で切り上げる。

**未知の既知 → プロトタイプ**
見た目・UXに関わるものは、実装前に単一HTMLファイルで複数案（3〜4方向の異なるアプローチ）のモックを作り、ユーザーに反応してもらう。バックエンドやステート管理は繋がず、fake data でよい。ユーザーの反応から好みを言語化してメモする。

**未知の未知 → ブラインドスポット・パス**
ユーザーの経験レベルを踏まえ、「考慮すらしていないであろう論点」を列挙して平易に説明する。コードベース・Web検索を活用してよい。各論点について「なぜ重要か」「今決めるべきか後回しでよいか」を添える。

**既知の既知** はそのまま尊重する。ただし矛盾や実現困難な点があれば指摘する。

### 4. 出力

最後に以下をまとめて提示する：

1. 判明した決定事項（インタビュー・プロトタイプ反応の結果）
2. 意図的に後回しにした未知（既知の未知として残すもの）
3. 実装に進む場合の推奨プロンプト（そのままコピペで次セッションに使える形）

## 注意

- このスキルは実装しない。未知を潰すのが役割。実装はユーザーの指示を待つ。
- 全象限を毎回フルに回す必要はない。タスクの規模に応じて省略し、省略した旨を伝える。
- ユーザーはPMであり「見れば分かる」判断力は高い。言語化を代行するより、反応できる具体物（モック・選択肢）を早く出すことを優先する。
