# /quiz — マージ前の理解度クイズ

diff を眺めるだけでは「分かった気」になるだけで、挙動の多くは既存コードパスに依存する。このスキルはレビュアー不在の個人開発において、マージ前にユーザー自身の理解を検証する。

## 手順

### 1. 対象の特定

引数で指定がなければ、直近の変更（未コミットの diff、なければ直近のコミット群やブランチ差分）を対象にする。`git diff` / `git log` で範囲を確定し、対象をユーザーに明示する。

### 2. 変更の深掘り

diff だけでなく、**変更が依存・影響する既存コードパス**まで読む。「この変更が既存の何と絡むか」がクイズの価値の源泉。

### 3. HTMLレポートの生成

単一HTMLファイルをスクラッチパッドに作り、Artifact として公開する（不可ならファイル送付）。構成：

1. **背景と意図** — なぜこの変更をしたか
2. **直感的な説明** — 技術的詳細の前に、何がどう変わったかを平易に
3. **変更内容** — ファイルごとの要点。既存コードとの接点を必ず含める
4. **リスクと注意点** — 壊れうる箇所、暗黙の前提、逸脱があればそれも
5. **クイズ（5〜8問）** — ページ末尾に配置

### 4. クイズの設計基準

- diff を眺めただけでは答えられない問題を最低2問入れる（既存コードパスとの相互作用、エッジケースでの挙動など）
- 「この変更後、Xという操作をしたら何が起きるか」形式の挙動予測問題を含める
- 選択式を基本とし、クリックで正誤と解説が表示される形にする（JS はインライン）
- ひっかけではなく、理解の穴を見つけるための問題にする

### 5. 採点とマージ判定

ユーザーの回答を受けたら採点し：

- **満点** → マージ・コミットに進んでよい旨を伝える
- **不正解あり** → 間違えた箇所を丁寧に解説し、関連コードを提示する。理解が追いついたら再クイズ（別の問題）を出す。**満点になるまでマージを勧めない**

## 注意

- レポートは日本語で書く。
- 変更が小さい（数行・自明）場合は、クイズが過剰である旨を伝えて簡潔な口頭サマリで代替してよい。
